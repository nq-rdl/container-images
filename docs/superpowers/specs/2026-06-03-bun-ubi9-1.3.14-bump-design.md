# bun-ubi9: bump Bun 1.1.42 → 1.3.14

- **Date:** 2026-06-03
- **Status:** Approved (design)
- **Branch:** `feat/bun-ubi9-1.3.14`
- **Closes:** Issue #39 (*feat(bun-ubi9): bump Bun to >=1.2 (target 1.3.x) — text bun.lock unsupported on 1.1.42*)

## 1. Problem

`ghcr.io/nq-rdl/bun-ubi9` ships **Bun 1.1.42**. The downstream `nq-rdl/website`
build (branch `feat/dockerfile-catalog-images`) is migrating off Docker Hub's
`oven/bun` onto this catalog image and fails on the first `bun install`: its text
`bun.lock` uses `lockfileVersion: 1`, a format introduced in **Bun 1.2**, which
1.1.42 cannot parse (`InvalidLockfileVersion: Unknown lockfile version`). The
website targets the Bun 1.3 line (`@types/bun: ^1.3.9`); its developers run 1.3.4
locally.

## 2. Goals / Non-goals

**Goals**

- Bump `images/bun-ubi9` to **Bun 1.3.14** (latest 1.3.x at time of writing).
- Keep the `ubi9/ubi-minimal:9.5` base and its `@sha256:` digest pin.
- Keep the existing download + checksum-verify mechanism in the Containerfile.
- Keep `linux/amd64` + `linux/arm64` platforms and the tag/digest conventions.

**Non-goals**

- Changing the checksum mechanism (e.g., git-pinning the SHA-256). The issue
  asks to keep the current verification as-is. Noted trade-off in §6.
- Touching `CONTRIBUTING.md`'s generic `e.g., 1.1.42` tag-convention example or
  the immutable `docs/superpowers/*` history.
- Any change to `support.eol` (`2026-12-31`) or `support.status` (`stable`).

## 3. Why 1.3.14

The issue asks for "≥1.2; ideally the latest 1.3.x". 1.3.14 is the newest 1.3
patch and was verified reachable for both target arches with valid checksums:

| Asset | HTTP | sha256 (from release `SHASUMS256.txt`) |
|-------|------|-----------------------------------------|
| `bun-linux-x64.zip` | 200 | `951ee2aee855f08595aeec6225226a298d3fea83a3dcd6465c09cbccdf7e848f` |
| `bun-linux-aarch64.zip` | 200 | `a27ffb63a8310375836e0d6f668ae17fa8d8d18b88c37c821c65331973a19a3b` |
| `SHASUMS256.txt` | 200 | — |

The Containerfile downloads `SHASUMS256.txt` from the same release and runs
`sha256sum -c`, so these hashes are *verified at build time*, not hard-coded.

## 4. Changes

| File | Edit |
|------|------|
| `images/bun-ubi9/Containerfile` | `ARG BUN_VERSION=1.1.42` → `1.3.14`. The `case`/curl/`sha256sum -c` logic and the base digest are version-agnostic and unchanged. The `LABEL …description` interpolates `${BUN_VERSION}`, so it updates automatically. |
| `images/bun-ubi9/image.yaml` | `runtime.version: "1.1.42"` → `"1.3.14"`; `tags: ["1.1","1.1.42","latest"]` → `["1.3","1.3.14","latest"]`. |
| `images/bun-ubi9/README.md` | "Supported tags" `1.1.42`/`1.1` → `1.3.14`/`1.3`; details table `Runtime \| Bun 1.1.42` → `Bun 1.3.14`. |
| `.changes/unreleased/Changed-*.yaml` | New changie fragment, kind **Changed** (a runtime minor-line bump is a consumer-visible behavior change). |

### Two sources of truth (must move together)

`bun-ubi9`'s `image.yaml` has no `build_matrix`, so CI builds with the
Containerfile's **default** `ARG BUN_VERSION` — that bakes the binary.
`image.yaml`'s `tags:`/`platforms:` only *label and fan out* the result. Bumping
one without the other would publish a `1.3.14` tag over a `1.1.42` binary.

## 5. Verification (before PR)

1. `podman build --build-arg BUN_VERSION=1.3.14 --platform linux/amd64 …` — the
   in-build `sha256sum -c` and final `bun --version` must pass and print `1.3.14`.
2. `podman run --rm <image> --version` — honors the `smoke-cmd` (`--version`)
   contract used by `scripts/smoke-test.sh`.
3. `pixi run policy-check-image-meta` (tag pattern `X.Y.Z`/`X.Y`/`latest`),
   `policy-check-containerfiles`, and `lint-containerfiles` (hadolint) — green.
4. `linux/arm64` is left to CI's multi-arch build; its asset + checksum were
   verified reachable (§3), and the Containerfile path is arch-symmetric.

## 6. Risks / trade-offs

- **Checksum trust model (unchanged):** the zip and its `SHASUMS256.txt` come
  from the same GitHub release, so verification is trust-on-fetch rather than
  git-pinned. Kept per the issue; a future hardening could pin the SHA-256 in the
  Containerfile so the integrity anchor lives in version control.
- **Major-line jump (1.1 → 1.3):** consumers pulling the moving `latest`/minor
  tags get Bun 1.3 behavior. This is the intent (it unblocks the website) and is
  recorded as a **Changed** changelog entry.

## 7. Rollout

Branch `feat/bun-ubi9-1.3.14` → PR (`Closes #39`). On merge to `main`, `build.yml`
rebuilds and re-points `latest`, `1.3`, and `1.3.14` to the new multi-arch digest.
The website then pins the published digest via its `ARG BUN_IMAGE`.
